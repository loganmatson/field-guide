Part 02 · Developer Setup

Set up Claude Code

Claude that lives in your codebase — installs in the terminal, plugs into VS Code and GitHub, and writes and runs real code. This guide takes you from zero to your first accepted change.

🧑‍💻
Think of Claude Code as a senior developer who lives inside your codebase. You describe what you want; they read every relevant file, write the code, run it, and hand you a diff to approve — like a teammate who already read the whole repo before you asked.
💬 Claude.ai

You paste code in, copy answers out. No file access, no terminal, no awareness of your project.

⚡ Claude Code (this guide)

Runs in your terminal and VS Code. Reads your files, edits them, runs commands, and knows your git history — all with your approval.

Step 1

Install the Claude Code CLI

Two small things first, then one command. Reveal each command when you're ready to run it.

1.1 Make sure you have Node.js

Claude Code installs through npm, which comes with Node.js. Check if you already have it:

terminal
$ node --version
If you see a version number like v20.x.x, you're set. If you see "command not found," install Node.js from nodejs.org (pick the LTS version) and run the check again.
1.2 Install Claude Code

One command installs it everywhere on your machine:

terminal
$ npm install -g @anthropic-ai/claude-code
The -g means "global" — it installs Claude Code once, for use in any folder on your computer. After this you can type claude from anywhere.
⚠️
Common mistake: closing the terminal before the install finishes. It can take a minute. Wait until you get your prompt ($) back before moving on.
1.3 Connect your account

Get an API key from console.anthropic.com → API Keys → Create Key. Then start Claude Code — it'll walk you through signing in the first time:

terminal
$ claude
Set ANTHROPIC_API_KEY in your shell profile (~/.zshrc on Mac) and you'll never have to paste it again — every new terminal picks it up automatically.
Step 2

Add the VS Code extension

The terminal is the engine. The VS Code extension is the dashboard — same Claude, friendlier view.

2.1 Install from the marketplace

In VS Code, open Extensions (the squares icon in the left bar), search "Claude Code", and click Install. A Claude panel appears in your sidebar.

2.2 Open a project folder

File → Open Folder → pick the project you want to work on. Claude Code now sees every file in that folder and can read, edit, and run things — always showing you a diff before it changes anything.

🖥️
Terminal vs. extension is like command line vs. control panel. Same power underneath — the extension just gives you buttons, inline diffs, and clickable file links instead of typing everything.
Use the VS Code extension while you're learning — the inline diffs make it obvious what Claude is doing. Drop to the terminal for long-running tasks and when you're working over SSH on a remote machine.
Step 3

Your first project, end to end

The whole loop: ask → review → approve. Here's how one task actually flows.

  • Describe the task in plain English — "Add a dark-mode toggle to the settings page."
  • Claude reads the relevant files on its own and proposes the change.
  • You review the diff — green is added, red is removed, just like code review.
  • Approve or reject. Nothing touches your files until you say yes.
  • Claude can run it — tests, the dev server — and fix what breaks, again with your approval.
👀
Reviewing Claude's diff is like code review with a coworker who already read the whole repo. You're not writing — you're approving. The judgment stays yours; the typing doesn't.
⚠️
Common mistake: approving a big diff without reading it. Start small. Read every diff for your first few tasks until you trust the pattern — then you can move faster.
Step 4 · The one people get wrong

Memory — how Claude Code remembers

There's no cloud "brain" to configure. In Claude Code, memory is just files on your machine — which means you own it, and it works offline. It comes in three stacking layers.

🗂️
The mental model people get wrong: they picture memory as a database in the cloud. It's the opposite. A folder on your disk is the memory. Whatever lives in that folder, Claude can see — and nothing leaves your machine except what's needed to answer the task in front of it.

Three layers, each more powerful than the last. Climb as far as you want — Layer 1 needs zero setup, Layer 3 builds itself over time.

Layer 1 · zero setup
The folder is the memory
Point Claude at a folder; its files are the working context.
Layer 2 · one file
CLAUDE.md
A standing brief that loads every session — who you are, the rules.
Layer 3 · self-building
Auto-memory
Claude writes its own memory files and recalls them across sessions.
📁 The project folder is the memory
Effort: none — make a folder · Payoff: durable context with zero configuration

This is the whole foundation. When Claude Code runs inside a folder, everything in it — your files, notes, prior work — is the working memory. Close your laptop, reopen tomorrow, point Claude at the same folder, and it picks up right where you left off, because the files are still there.

1 Make a folder

Anywhere you like — your Desktop is fine. Name it for the work, e.g. a folder for one client, one project, or one area of your business.

terminal
$ mkdir ~/Desktop/MyProject
$ cd ~/Desktop/MyProject
2 Drop your context in

Put whatever Claude should know into the folder — reference docs, your templates, notes, past work. These files become the context it reads before answering.

3 Open it and start

In VS Code: File → Open Folder → pick it. Or in the terminal, run claude from inside the folder. That's it — the folder is now the brain.

⚠️
Common mistake: thinking you need a database or an account setting. You don't. The folder is the memory. One folder per project keeps each one's context clean and separate.
📋 CLAUDE.md — the file that makes it "remember who you are"
Effort: one file, a few minutes · Payoff: persistent identity loaded on every single run

A plain text file named CLAUDE.md in your folder. Claude Code reads it automatically at the start of every session — so it never forgets your role, your rules, or your house style. This is the one file that survives forever and makes the agent feel like it works for you.

1 Create CLAUDE.md in your folder

Just a markdown file at the root of the folder. Claude finds it on its own — no setting to flip.

2 Fill in this starter template

Adapt it to you. Keep it tight — it loads every time, so signal beats volume:

CLAUDE.md
# Who I am
- I'm a [role] and this folder is for [what this project is]
- My audience / clients are [who]

# How to work with me
- Always show me options before taking an action
- Format answers as [short / bullets / full prose]
- Flag anything [legal/regulatory/confidential] for my review

# House rules
- Never [the thing] without asking first
- Match the voice and structure of my existing files
3 Save and test

Save it, then ask Claude something normal. It should answer in your format and respect your rules without being reminded — that's CLAUDE.md doing its job.

Layer 1 (the folder) gives Claude your material. Layer 2 (CLAUDE.md) gives it your instructions — the standing rules that apply no matter what you're working on. On a team, commit CLAUDE.md to the repo and every teammate inherits the same setup automatically (that's Step 6).
🌱 Auto-memory — the memory that writes itself
Effort: set a directory once · Payoff: Claude learns across sessions and compounds over time

This is the real differentiator from a wrapped chatbot. Beyond reading your files, Claude Code can write its own memory — saving durable facts to disk during one session and recalling them in the next. Over weeks it accumulates a knowledge base about your work that you never had to type.

1 How it's structured

Memory lives as one fact per file inside a memory/ folder, with a MEMORY.md index that loads each session. Claude reads the index to know what it already knows, then opens the specific file when a fact is relevant:

MyProject/ ├── CLAUDE.md # Layer 2 — your standing rules └── memory/ ├── MEMORY.md # the index — one line per fact ├── client-acme-prefs.md ├── project-q3-goals.md └── feedback-tone.md # Claude wrote these itself
2 Turn it on

Point Claude at a memory directory once. The simplest way: add a line to your CLAUDE.md telling it to maintain memory, and create the folder:

terminal
$ mkdir memory
# then add to CLAUDE.md:
# "Maintain a persistent memory in ./memory/ —
#  one fact per file, indexed in MEMORY.md.
#  Save durable facts; recall them each session."
3 Let it build

Now, when something is worth remembering — a client's preference, a decision you made, feedback you gave — Claude writes it to a memory file and notes it in the index. Next session, it reads the index first and already knows. You can open any file and read, edit, or delete what it saved; it's all plain text you control.

📈
This is the layer that compounds. A chatbot starts cold every time. With auto-memory, month three is dramatically better than week one — the agent has quietly built a file cabinet of everything it has learned about your work, and every file is yours to read.

The reason this scales without blowing up the context window is a deliberate two-tier split. Some things load automatically, every session. The expensive details load only when they become relevant.

Tier A — injected at session startalways loaded
📄
CLAUDE.md (global + project)
Full contents, stacked. This file is the switchboard — whatever it links gets pulled in with it.
📐
rules/*.md — how to behave
Loaded because CLAUDE.md imports them. Cut the link and a rule stops loading. (A separate harness layer may also inject its own bundle — see the proof below.)
🗂️
MEMORY.md — the index only
Who you are / what you're working on — but tagged as background context, "true when written," not as a command. Memory can drift, so it's read, then verified before acting on.
⏱️
Session reminder
Today's date, your account, the working folder, the available skills — and the MEMORY.md index a second time.
Tier B — opened on demand, mid-sessionloaded only if relevant
🌱
memory/<specific-fact>.md
The individual fact files behind the index stay on disk. Claude reads the index to know what it knows, then opens a single file only when that fact actually comes up. The index is the cheap pointer; the facts are the expensive payload, fetched just-in-time.
The third ring — context, the session's live working memorythis session only

Persistent memory (above) lives on disk forever. But within a single session there's a second, faster kind of memory: the context window — everything Claude is holding right now in this conversation. It's where the loaded files, your messages, and Claude's replies all live while you work. Two things to know about it:

📏
It's finite
The window has a ceiling. Everything loaded this session — files, history, replies — shares that space. It's working memory, not storage; close the session and it's gone (which is why the on-disk layers above exist).
🗜️
When it fills, it compacts
Rather than forgetting or cutting off, Claude summarizes the older part of the conversation into a compact form and keeps going — so a long session doesn't hit a wall. You can steer what survives a compaction with a rule (e.g. "always preserve the list of files changed and any in-progress task"). The summary carries forward; the work continues uninterrupted.
🧠
Think of it as desk vs. filing cabinet. The context window is the desk — fast, limited, cleared each day. The on-disk layers are the filing cabinet — slower to reach for, but permanent. Compaction is tidying the desk mid-task: it sweeps the clutter into a neat summary so there's room to keep working, without throwing anything important away.
Verified live, this session In one real workspace, the rules/ folder held 5 files — but the project's CLAUDE.md linked only 3 of them, and only those 3 loaded; the other 2 sat unread on disk. The memory index pointed to 47 fact files, and zero of them loaded until one was needed. That's the whole trick: a lightweight index always on, a deep archive pulled just-in-time, and a context window that compacts itself when full — so memory can grow for months without ever filling the window.
💡
Why this beats a fixed prompt: a wrapped chatbot stuffs everything it "knows" into one static system prompt — it doesn't grow, and eventually it overflows. This split lets the knowledge base expand indefinitely on disk while only the relevant slice ever enters the conversation.
Step 5 · The big one

GitHub integration

This is where Claude Code goes from "handy" to "part of the team." It reads your history, works in branches, and reviews pull requests.

📁Your repolocal + GitHub
→
🔎Claude readsfiles + git history
→
✏️Suggests changeon a branch
→
✅You approvereview the diff
→
🚀Commit / PRpushed to GitHub
5.1 Make sure your repo is a git repo

If you cloned it from GitHub, it already is. If not, initialize it once:

terminal
$ git init
$ git remote add origin https://github.com/you/your-repo.git
5.2 Let Claude work in branches

Ask Claude to make a change and it can create a branch, commit its work, and push — so your main branch stays clean. You review the branch before merging, exactly like a teammate's PR.

Branches keep experiments off your main code. Claude does its work in a side branch; if you like it, you merge; if not, you delete the branch and nothing's lost. It's a safety net built into how git already works.
5.3 Code review on pull requests

Connect Claude Code to GitHub and it can review open PRs — leaving inline comments on specific lines, catching bugs, and suggesting fixes. It reads the surrounding code and git history, so its comments are about your codebase, not generic advice.

The lowest-risk way to introduce Claude Code to a team is as a PR reviewer. It never merges anything — it just comments. Everyone gets value immediately, and trust builds before you let it write code directly.
⚠️
Common mistake: giving Claude access to a repo without committing your current work first. Commit or stash what you have before a big task, so you always have a clean point to roll back to.
Step 6

Run Claude Code on a team

The trick to a team setup: everyone inherits the same instructions automatically, straight from the repo.

6.1 Add a shared CLAUDE.md

Drop a CLAUDE.md file in the root of your repo. Claude Code reads it automatically on every run. Put your conventions, architecture notes, and do/don't rules in it — and because it's committed to the repo, every teammate gets the same setup for free.

CLAUDE.md
# Project conventions
- Use TypeScript, no `any`
- Tests live next to the file, named *.test.ts
- Never edit /generated — it's auto-built
- Ask before adding a new dependency
6.2 Folder-level instructions

You can add a CLAUDE.md inside specific folders too — handy for a monorepo where each package has its own rules. Claude reads the closest one to whatever it's working on.

📋
A committed CLAUDE.md is your team's shared playbook. New hire clones the repo and their Claude already knows the house style — no onboarding doc to read, no tribal knowledge to absorb.

🙋 Questions you'll have

Tap any question.

🔎 Does Claude see my whole codebase? +
It can read files in the folder you opened, but it pulls in what's relevant to the task rather than uploading everything blindly. You stay in control of which folder it's pointed at, and you can scope it further with config.
✋ What if it edits something I didn't want? +
Every change is shown as a diff you approve before it's applied — nothing is silent. And because you work in git, you can always undo: git checkout . reverts unwanted edits, and branches keep experiments off your main code.
👥 Can multiple devs use it on the same repo? +
Yes — that's the point of the committed CLAUDE.md. Everyone runs their own Claude Code locally, but they all inherit the same project instructions from the repo, so the team works to one consistent standard.
🧠 Where does memory actually live — and is confidential data safe? +
All three memory layers are plain files on your own machine — the folder, CLAUDE.md, and the memory/ directory. Nothing is stored in a Claude "cloud brain"; only the slices needed to answer a given task are sent to the model. That's a feature if you handle confidential work — your context stays under your control. One gotcha to flag: if your company syncs your Desktop or Documents to OneDrive or Microsoft Teams/SharePoint, your project folder (and its memory files) may sync to your corporate cloud — a copy outside Claude. Usually fine, but if you advise clients under confidentiality, keep that folder somewhere that isn't auto-synced, or confirm your firm's data-handling policy first.
🔐 Is my code sent to Anthropic? +
Relevant code is sent to the model to answer your request, the same way any AI coding tool works. On business plans your code isn't used to train models, and there are admin controls for teams. For exact terms, point to Anthropic's data-handling docs — and check whether your org wants the business/enterprise tier for tighter guarantees.
