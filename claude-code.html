Part 02 · Developer Setup

Set up Claude Code

Claude that lives in your codebase — installs in the terminal, plugs into VS Code and GitHub, and writes and runs real code. This guide takes you from zero to your first accepted change.

🧑‍💻
Think of Claude Code as a senior developer who lives inside your codebase. You describe what you want; they read every relevant file, write the code, run it, and hand you a diff to approve — like a teammate who already read the whole repo before you asked.
💬 Claude.ai

You paste code in, copy answers out. No file access, no terminal, no awareness of your project.

⚡ Claude Code (this guide)

Runs in your terminal and VS Code. Reads your files, edits them, runs commands, and knows your git history — all with your approval.

Step 1

Install the Claude Code CLI

Two small things first, then one command. Reveal each command when you're ready to run it.

1.1 Make sure you have Node.js

Claude Code installs through npm, which comes with Node.js. Check if you already have it:

terminal
$ node --version
If you see a version number like v20.x.x, you're set. If you see "command not found," install Node.js from nodejs.org (pick the LTS version) and run the check again.
1.2 Install Claude Code

One command installs it everywhere on your machine:

terminal
$ npm install -g @anthropic-ai/claude-code
The -g means "global" — it installs Claude Code once, for use in any folder on your computer. After this you can type claude from anywhere.
⚠️
Common mistake: closing the terminal before the install finishes. It can take a minute. Wait until you get your prompt ($) back before moving on.
1.3 Connect your account

Get an API key from console.anthropic.com → API Keys → Create Key. Then start Claude Code — it'll walk you through signing in the first time:

terminal
$ claude
Set ANTHROPIC_API_KEY in your shell profile (~/.zshrc on Mac) and you'll never have to paste it again — every new terminal picks it up automatically.
Step 2

Add the VS Code extension

The terminal is the engine. The VS Code extension is the dashboard — same Claude, friendlier view.

2.1 Install from the marketplace

In VS Code, open Extensions (the squares icon in the left bar), search "Claude Code", and click Install. A Claude panel appears in your sidebar.

2.2 Open a project folder

File → Open Folder → pick the project you want to work on. Claude Code now sees every file in that folder and can read, edit, and run things — always showing you a diff before it changes anything.

🖥️
Terminal vs. extension is like command line vs. control panel. Same power underneath — the extension just gives you buttons, inline diffs, and clickable file links instead of typing everything.
Use the VS Code extension while you're learning — the inline diffs make it obvious what Claude is doing. Drop to the terminal for long-running tasks and when you're working over SSH on a remote machine.
Step 3

Your first project, end to end

The whole loop: ask → review → approve. Here's how one task actually flows.

  • Describe the task in plain English — "Add a dark-mode toggle to the settings page."
  • Claude reads the relevant files on its own and proposes the change.
  • You review the diff — green is added, red is removed, just like code review.
  • Approve or reject. Nothing touches your files until you say yes.
  • Claude can run it — tests, the dev server — and fix what breaks, again with your approval.
👀
Reviewing Claude's diff is like code review with a coworker who already read the whole repo. You're not writing — you're approving. The judgment stays yours; the typing doesn't.
⚠️
Common mistake: approving a big diff without reading it. Start small. Read every diff for your first few tasks until you trust the pattern — then you can move faster.

GitHub integration

4.1 Make sure your repo is a git repo

If you cloned it from GitHub, it already is. If not, initialize it once:

terminal
$ git init
$ git remote add origin https://github.com/you/your-repo.git
4.2 Let Claude work in branches

Ask Claude to make a change and it can create a branch, commit its work, and push — so your main branch stays clean. You review the branch before merging, exactly like a teammate's PR.

Branches keep experiments off your main code. Claude does its work in a side branch; if you like it, you merge; if not, you delete the branch and nothing's lost. It's a safety net built into how git already works.
4.3 Code review on pull requests

Connect Claude Code to GitHub and it can review open PRs — leaving inline comments on specific lines, catching bugs, and suggesting fixes. It reads the surrounding code and git history, so its comments are about your codebase, not generic advice.

The lowest-risk way to introduce Claude Code to a team is as a PR reviewer. It never merges anything — it just comments. Everyone gets value immediately, and trust builds before you let it write code directly.
⚠️
Common mistake: giving Claude access to a repo without committing your current work first. Commit or stash what you have before a big task, so you always have a clean point to roll back to.
Step 5

Run Claude Code on a team

The trick to a team setup: everyone inherits the same instructions automatically, straight from the repo.

5.1 Add a shared CLAUDE.md

Drop a CLAUDE.md file in the root of your repo. Claude Code reads it automatically on every run. Put your conventions, architecture notes, and do/don't rules in it — and because it's committed to the repo, every teammate gets the same setup for free.

CLAUDE.md
# Project conventions
- Use TypeScript, no `any`
- Tests live next to the file, named *.test.ts
- Never edit /generated — it's auto-built
- Ask before adding a new dependency
5.2 Folder-level instructions

You can add a CLAUDE.md inside specific folders too — handy for a monorepo where each package has its own rules. Claude reads the closest one to whatever it's working on.

📋
A committed CLAUDE.md is your team's shared playbook. New hire clones the repo and their Claude already knows the house style — no onboarding doc to read, no tribal knowledge to absorb.

🙋 Questions you'll have

Tap any question.

🔎 Does Claude see my whole codebase? +
It can read files in the folder you opened, but it pulls in what's relevant to the task rather than uploading everything blindly. You stay in control of which folder it's pointed at, and you can scope it further with config.
✋ What if it edits something I didn't want? +
Every change is shown as a diff you approve before it's applied — nothing is silent. And because you work in git, you can always undo: git checkout . reverts unwanted edits, and branches keep experiments off your main code.
👥 Can multiple devs use it on the same repo? +
Yes — that's the point of the committed CLAUDE.md. Everyone runs their own Claude Code locally, but they all inherit the same project instructions from the repo, so the team works to one consistent standard.
🔐 Is my code sent to Anthropic? +
Relevant code is sent to the model to answer your request, the same way any AI coding tool works. On business plans your code isn't used to train models, and there are admin controls for teams. For exact terms, point to Anthropic's data-handling docs — and check whether your org wants the business/enterprise tier for tighter guarantees.
