01Setup

You'll have Claude Code installed and running in your own terminal in under three minutes. Then we'll show you where things live, where to write them, and why a tool that reads your folder is worth keeping there.

→ Before you go anywhere, jump to Section 06 — Ten Takeaways — at least once. It's the shortest path to getting real use out of this card.
The thesis

Claude Code is the practice of taking the parts of your week that already have a shape — the routines, the checks, the briefs you write the same way every Monday — and turning them into named, repeatable instructions a tool can run for you.

That matters because the cost of doing the same thing by hand isn't the fifteen minutes a day — it's the cognitive drag of remembering to do it, the mistakes that creep in when you're tired, and the senior teammates who never get back the time you'd reclaim if it didn't fall on you.

The work compounds: every skill you name is a permanent asset on your laptop, every skill you connect to a real system through MCP is a force multiplier, and the version of you six months from now is built from the small workflows you bothered to write down today.

i
Install (Mac · Linux · WSL)
curl -fsSL https://claude.ai/install.sh | bash
One line. No node, no docker, no setup wizard. Windows users: winget install Anthropic.ClaudeCode.
ii
Launch inside a project folder
cd ~/your-project && claude
Claude reads the folder you launch from. It already knows the codebase you're standing in — don't paste files, just point.
iii
Your first prompt — read, don't write
explain what this project does and how the pieces fit together
Resist the urge to ask it to code. Ask it to read. The quality of every later prompt depends on Claude already having the lay of the land.
Surface A

The terminal

The original way to use Claude Code. Open Terminal (Mac) or Windows Terminal, navigate to your project, type claude. Conversation lives in the terminal window — type prompts, see responses, hit Ctrl+C to interrupt.

  • Best for — quick tasks, scripting, running on a remote machine over SSH
  • Watch out for — long conversations scroll off the screen; use /resume to come back to a session
Surface B

The VS Code extension

Install the Claude Code extension from the VS Code marketplace. Opens as a panel alongside your editor. Same Claude, same skills, same files — you just get a UI with diff previews, file-tree integration, and Cmd+Shift+P access.

  • Best for — building skills, editing files Claude proposes, reviewing diffs before accepting
  • Watch out for — keyboard shortcuts differ slightly from the terminal; both call the same engine underneath
Where things live

Project layout you'll grow into

~/.claude/skills/<name>/SKILL.md
Personal skills. Available in every project you open. Put cross-project workflows here.
./.claude/skills/<name>/SKILL.md
Project skills. Live inside the repo, shared with teammates via git. Use for project-specific workflows.
~/.claude/CLAUDE.md
Personal standing instructions. Conventions Claude applies to every project — your name, your style, your defaults.
./CLAUDE.md
Project house rules. Checked into git. Team-shared coding standards, libraries to use, what to never do.
./CLAUDE.local.md
Personal overrides for this project. Gitignored. Where you put your own preferences that shouldn't ship with the repo.
./.claude/settings.json
Hooks, permissions, env vars. Lifecycle automation lives here.

Section 05  ·  what gets stored where · what stays on your laptop

05Memory & safety

Claude Code reads files on your laptop. That's the whole architecture. Three things live in that read path — and each scope has a use, a constraint, and a trap.

Scope 1 · Project
./CLAUDE.md
Lives inside the repo. Committed to git. Shared with everyone on the team. Use for: coding standards, libraries, conventions, "never do X here." If you'd put it in a README, put it here.
Scope 2 · User
~/.claude/CLAUDE.md
Lives on your laptop. Never shared. Use for: your name, your preferred tone, defaults across every project, "I work in Eastern time." Personal, not project.
Scope 3 · Local
./CLAUDE.local.md
Inside the project but gitignored. Use for: your personal overrides on a shared project — paths only you have, debugging notes, the things you'd never put in the team's PR.

Importing one file from another

CLAUDE.md can pull in any other markdown file using the @import syntax. The contents land in Claude's context as if they were written inline. Use this to keep CLAUDE.md short and reference longer style guides separately.

# House rules — pricing model All cash math uses Decimal, never float. Round per banker's rounding (ROUND_HALF_EVEN). See @docs/style-guide.md for the full convention set. See @docs/api-contracts.md before touching the trade API.
Compliance lens · what NOT to put in memory

Treat CLAUDE.md like a Slack channel

×
No API keys, passwords, or tokens.Memory files are plain text on disk. Anything that ends up in CLAUDE.md gets read by Claude on every session — and ends up in your terminal logs.
×
No proprietary models, alpha signals, or position-level data.If a teammate pulls the repo, they see your project CLAUDE.md. Don't bury your edge in a shared instruction file.
×
No client names, account numbers, or PII.Same rule as Slack. If you wouldn't post it in #general, don't write it in CLAUDE.md. The compliance posture is identical.
×
No "for my eyes only" content in shared scopes.Use CLAUDE.local.md for personal notes. The .local.md suffix is gitignored by default — anything you'd be embarrassed to share belongs there.
