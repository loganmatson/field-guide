Claude Code · Team Setup Guide

One team, one shared box of skills.

This manual walks you through getting Claude Code running, plugging into your team's shared skill library, and sharing your own work back. No prior coding needed. Copy, paste, done.

Any organizationCopy-paste ready~20 min setup
!
Commands are written for Windows.

Every command below assumes a Windows PC using PowerShell. If anyone is on a Mac, the ideas are identical but a few commands differ — check the official Claude Code docs for the Mac equivalents before following the install steps.

★

How sharing actually works

Read this first. If you understand these three ideas, nothing later will surprise you.

THE LIBRARY

One shared place online (a GitHub repo) that holds the team's official skills. This is the single source of truth.

→
YOUR COPY

When you install a skill, a copy lands on your own PC. You run that copy. It can drift from the library if you edit it.

Three ideas to keep in your pocket

  • There are always two copies of a skill — the official one in the library, and the one installed on your machine.
  • You share files, never access — the library hands out skill files. It does not hand out your logins. Everyone signs in as themselves.
  • "Pushed" is not "received" — when the lead updates a skill, you only get it after you run the update command. Nothing arrives automatically.

Analyst track · the five steps everyone follows

1

Install Claude Code

First we put the tool on your PC, then we add it to VS Code so you can use it while you work.

Install the tool itself

Open the Start menu, type PowerShell, and open it. Paste this one line and press Enter.

PowerShell
irm https://claude.ai/install.ps1 | iex

That's it. No need to install anything else first.

✓
Prefer clicking to typing?

You can instead run winget install Anthropic.ClaudeCode in PowerShell, which is Windows' built-in app installer. Either route gives you the same tool.

Recommended: install Git for Windows

Git is the engine behind the shared library. Installing it lets Claude Code work at full power instead of falling back to a limited mode. Download it from the official site and click through the installer with the default options.

Download page
https://git-scm.com/download/win

Add it to VS Code

VS Code is the workspace where you'll actually use Claude Code. If you don't have it yet, install it first.

  • Download VS Code from https://code.visualstudio.com and install it.
  • Open VS Code, click the Extensions icon on the left sidebar (the four little squares).
  • Search Claude Code, find the one published by Anthropic, and click Install.

You now have Claude Code living right inside your editor.

2

Sign in to your team account

Your access is personal. The library shares skill files with everyone, but each person signs in with their own account.

Open Claude Code and log in

  • In VS Code, open the Command Palette with Ctrl + Shift + P.
  • Type Claude Code and pick the option to open it.
  • A browser window opens. Sign in with the Claude account your lead invited you to, then come back to VS Code.
!
Didn't get an invite yet?

You need a seat on the team plan before this step will work. If sign-in says you have no access, message your lead, who handles invites in step 1 of the Lead track.

3

The skills folder, done safely

This is the step people get wrong most. Skills have one correct home on your PC. Put them anywhere else and Claude Code simply won't see them.

Think of it like one labeled shelf in your house

Claude only ever looks in one shelf for your personal skills. That shelf is a folder named .claude inside your user folder. Every skill is a small box that sits on that shelf.

If you build a beautiful skill but set the box down on the kitchen counter instead of the shelf, Claude walks right past it. It isn't broken, it's just in a place Claude never checks. The whole trick is putting every box on the same shelf the team agreed on.

Keep your shelf tidy and matching the team's, and your tools and theirs stay in sync. Scatter boxes around the house, and things quietly go missing.

Where the shelf lives on Windows

Your personal skills live here, where <you> is your Windows username.

Folder path
C:\Users\<you>\.claude\skills\

Inside that skills folder, each skill is its own little folder with a file named SKILL.md inside it.

What one skill looks like
.claude\
  skills\
    interview\
      SKILL.md
    darth-vader\
      SKILL.md

Can't see the .claude folder?

Folders that start with a dot are hidden by Windows. To reveal them, open File Explorer, click View at the top, then Show, then turn on Hidden items. The .claude folder will appear.

A shortcut: paste %USERPROFILE%\.claude straight into the File Explorer address bar and press Enter to jump there.

✕
Don't invent your own folders.

Making a new "Skills" folder on your Desktop or inside Documents feels organized, but Claude never looks there. Boxes off the shelf are invisible. Always use the one path above.

4

Connect to the team library

Now you point Claude Code at the team's shared library and pull in the official skills. Your lead will give you the real library name to use in place of the placeholder.

✎
Swap this in.

Everywhere you see your-org/team-skills, replace it with the real name your lead gives you, for example acme-corp/team-skills.

First, get access to the library

The library is a private repo, so Claude Code needs to know it's you. Run this once in PowerShell and follow the browser prompts.

PowerShell · run once
gh auth login

If gh isn't installed, grab it from https://cli.github.com, then run the line above.

Add the library and install its skills

Run these two lines inside Claude Code (in the Claude Code chat box in VS Code, not in PowerShell).

Inside Claude Code
/plugin marketplace add your-org/team-skills
/plugin install core-toolkit@team-skills

The shared skills now show up in your skill menu when you type /. You're connected.

Getting updates later

When your lead improves a skill, you don't get it automatically. Run this when they tell you there's an update.

Inside Claude Code
/plugin marketplace update team-skills
5

Share a skill back

Built something useful? Great. On this team, you propose it and the lead publishes it to the library. That keeps one clean, official version for everyone.

How to propose your skill

  • Find your skill folder on the shelf: C:\Users\<you>\.claude\skills\your-skill\.
  • Open SKILL.md and read it once for anything private, then send the whole folder to your lead.
  • Tell your lead in one line what it does and when it should fire. They review it and add it to the library.

Once it's published, the rest of the team picks it up with the same update command from step 4.

✕
Scrub before you send.

Skills sometimes pick up private notes, client names, or hard-coded paths from your PC. Read through it and remove anything personal before it goes to the shared library, even a private one.

!
A SKILL.md only has a few real settings.

At the top of the file, the only official settings are name, description, disable-model-invocation, and user-invocable. If you see invented ones like a triggers list, delete them, since Claude decides when to use a skill from its description.

Lead / Admin track · you set the library up once for everyone

1

Invite the team to the plan

Before anyone can sign in, they need a seat. This is the one step that happens on the website, not in code.

Add each person

  • Open your team's admin settings on the Claude website.
  • Invite each analyst by email so they each get their own seat.
  • Without a seat, their sign-in in Analyst step 2 will fail, so do this first.
2

Build the team library repo

The library is a GitHub repo with a specific shape. Get the shape right and Claude Code can install from it.

Create the folder structure

Run these in PowerShell to make the library's skeleton.

PowerShell
mkdir team-skills
cd team-skills
mkdir .claude-plugin
mkdir plugins\core-toolkit\skills

Add the library's table of contents

Create a file at .claude-plugin\marketplace.json with this inside. It lists what's in the library.

.claude-plugin\marketplace.json
{
  "name": "team-skills",
  "owner": { "name": "Your Team" },
  "plugins": [
    {
      "name": "core-toolkit",
      "description": "Standard skills everyone uses",
      "version": "1.0.0",
      "source": "./plugins/core-toolkit"
    }
  ]
}
!
Only the table of contents goes in .claude-plugin.

The skills themselves live in plugins\core-toolkit\skills\, not inside .claude-plugin. Putting skill folders in the wrong place is the most common setup mistake.

What the finished library looks like

Library layout
team-skills\
  .claude-plugin\
    marketplace.json
  plugins\
    core-toolkit\
      skills\
        interview\
          SKILL.md
        darth-vader\
          SKILL.md
3

Give the team access

Push the library to GitHub as a private repo, then add each analyst so they can reach it.

Push it private

PowerShell · inside team-skills
git init
git add .
git commit -m "Initial team skills library"
gh repo create your-org/team-skills --private --source=. --push

Add each analyst as a collaborator

A private repo is invisible to people you haven't added. Run this for each person, swapping in their GitHub username.

PowerShell
gh repo edit your-org/team-skills --add-collaborator their-username
✕
Skip this and step 4 silently fails.

If an analyst isn't a collaborator, their /plugin marketplace add returns "not found" with no clear reason. This is the number-one support ticket, so add everyone up front.

4

Curate the duplicate skills

Right now several analysts each built their own version of the same skills, like interview, tony-stark, and darth-vader. Your job is to turn each pile of near-duplicates into one official version.

For each duplicated skill

  • Collect everyone's version of that skill into one place.
  • Read them side by side and pick the strongest one, or merge the best parts into a single file.
  • Strip anything personal, any client names, and any hard-coded paths like C:\Users\someone\... that would break on other PCs.
  • Drop the winning version into plugins\core-toolkit\skills\ as the one true copy.

From now on, that single version is what the whole team installs. The private one-off copies retire.

✓
This is the moment drift stops.

Once a skill has one official home, "whose version is right" stops being a question. Everyone installs the same one and improvements flow through you.

5

Publish and push updates

Publishing a change is just saving it to the library and telling the team to pull it. Review analyst submissions the same way.

Ship a change

PowerShell · inside team-skills
git add .
git commit -m "Update interview skill"
git push

Then post one line to the team: run /plugin marketplace update team-skills. Until they do, they're still on the old version.

Reviewing what analysts send you

  • An analyst hands you a skill folder from Analyst step 5.
  • Read the SKILL.md: check it's clean, has no invented settings, and no machine-specific paths.
  • If it's good, drop it into the skills folder, commit, push, and announce the update.
✎

Fix a skill yourself, fast

A skill keeps tripping over the same thing? You can patch your own copy in seconds to unblock yourself, then send the real fix to your lead.

Edit your local copy in VS Code

  • In VS Code, choose File, then Open Folder, and open C:\Users\<you>\.claude\skills.
  • Find the skill's folder and click its SKILL.md to open it.
  • Make your edit, then save with Ctrl + S. The change is live right away on your machine.
!
A local fix only fixes your copy.

Editing your machine's copy unblocks you, but everyone else still has the old broken version. If the fix is worth keeping, send it to your lead so it reaches the whole team.

⚖

Who can change what

Anyone can change their own copy. Only the lead changes the official library copy. That split is what keeps a shared skill from quietly breaking for the whole team.

YOU, LOCALLY

Edit your installed copy anytime to unblock yourself. It affects only you and is meant to be temporary.

→
LEAD, OFFICIALLY

Takes your fix, reviews it, and publishes it to the library so everyone gets it on the next update.

The recurring-problem loop

  • Unblock now — patch your local copy so you can keep working.
  • Send it up — give the same fix to your lead with a one-line note on what kept failing.
  • Team pulls it — the lead publishes, posts the update line, and everyone runs the update.

If something fails over and over, don't just patch the symptom. Tell your lead the pattern so the official skill gets made sturdier for everyone.

⚑

Troubleshooting

The three things that actually go wrong, and the exact fix for each.

✕"Marketplace not found" when adding the library

Why: you either haven't run gh auth login yet, or your lead hasn't added you as a collaborator on the private repo.

Fix: run gh auth login first. Still stuck? Ask your lead to confirm you're a collaborator on the repo, then try the add command again.
✕A skill works for one person but fails for another

Why: the skill usually has a hard-coded path from someone's PC like C:\Users\jane\..., or it needs a connection that person set up and you didn't, such as Google or Slack.

Fix: open the SKILL.md and remove machine-specific paths so it works for everyone. Share the skill, never the personal login behind it.
✕A teammate is running an old version of a skill

Why: installing takes a snapshot. A push to the library does not reach anyone until they pull it.

Fix: have them run /plugin marketplace update team-skills inside Claude Code. Remember, "pushed" is not "received" until they update.
