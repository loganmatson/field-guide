04 · Where everything lives

The .claude/ folder, demystified.

Everything Claude Code uses sits in a .claude/ folder — in VS Code it shows up right in your file tree. Three scopes decide who sees what.

# in VS Code — open the file tree .claude/ ├── skills/ │ └── weekly-report/ │ ├── SKILL.md # the skill │ └── helper.py # optional ├── commands/ │ └── pre-pr.md # a /command ├── rules/ │ └── style.md # topic rules ├── settings.json # hooks + permissions └── CLAUDE.md # project memory
Global · you, everywhere
~/.claude/

Lives in your home folder. Skills, commands, and rules here work in every project on your machine.

Project · the team
./.claude/

Sits in the repo root. Check it in and every teammate inherits the same skills, commands, and CLAUDE.md.

Local · just you, this repo
CLAUDE.local.md · settings.local.json

Gitignored. Your personal overrides that stay off the team's shared setup.

05 · Your CLAUDE.md + memory

The file that gives Claude a memory.

Run /init once and Claude drafts a starter CLAUDE.md. It loads at the start of every session — so you stop re-explaining your project. This single file gives a stateless chatbot the memory of an assistant that knows your stack.

# CLAUDE.md — written once, loaded every session
- Monorepo: services in /backend, /web, /mobile
- Use pnpm, never npm
- Run pnpm test before every commit
- API errors return { code, message }
- Prefer small PRs; never push straight to main
Keep it tight · under ~200 lines. A bloated CLAUDE.md eats your context window and gets ignored. Put behavioral rules here; move long topic detail into .claude/rules/.
CLAUDE.mdyou write

Standards, architecture, conventions, "always do X." Behavioral guidance you control.

Auto-memoryClaude writes

Patterns Claude learns on its own — build quirks, debugging fixes — and reuses next session.
